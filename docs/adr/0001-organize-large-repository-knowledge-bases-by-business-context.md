# Organize large-repository knowledge bases by business context

For large repositories, the business knowledge base is organized by business context rather than by code directory or service module. The scheme is intended to be reusable across similar multi-service or multi-application codebases. Contexts are split by business capability boundaries: one business context may span multiple code directories, and one service may contribute to multiple business contexts. `CONTEXT.md` files are reserved for business language only; implementation facts such as routes, controllers, tables, MQ topics, and call graphs live in separate capability or evidence documents. Business capability documents use a fixed index-like structure and include frontmatter for domain and keyword indexing, so future agents can route work to the right knowledge document before opening code. Frontmatter status acts as a confidence signal: agents prefer `verified` documents during indexing and avoid treating `stale` or `deprecated` documents as authoritative. Capability documents require non-empty `confirmation_sources` before they can be treated as `verified`. Generated `INDEX.md` and `index.json` files accelerate human and agent retrieval and are committed with knowledge-base source changes, but Markdown knowledge documents remain the source of truth. The first plugin version focuses on knowledge-base maintenance, retrieval, skeleton initialization, indexing, archived PRD checklist and reference support, and write-boundary enforcement; code-fact extraction remains a pluggable evidence adapter rather than core plugin responsibility. Evidence documents are generated or semi-automatically refreshed from tools such as CodeGraph and repository scans; humans maintain summaries, business boundaries, design intent, and open questions rather than hand-maintaining long call chains. Knowledge is built incrementally from real work such as new requirements, archived PRDs, repeated questions, cross-module bugs, boundary confusion, or stale-code conflicts; archived requirements may trigger or supplement knowledge documents, but the knowledge base records only durable business and architecture knowledge rather than copying requirement tasks, reports, or full PRD content. When extracting from archived PRDs, capability documents keep final stable conclusions only; important rejected options or disputes become ADRs only when they have long-term architectural value. The scheme avoids one-time full extraction and empty placeholder contexts. New contexts are created only when a stable business language, clear boundaries, at least one real capability, and supporting evidence exist; otherwise agents should update an existing context or propose a candidate for review. Candidate documents may record recurring but unconfirmed business terms or possible domains, but candidates are not confirmed contexts and must not be treated as authoritative terminology or boundaries. Agents should read the knowledge-base rules before maintaining or routing through `docs/knowledge/**`, but ordinary code tasks do not need to enter this workflow unless they also update or rely on the knowledge base. Agents should use the knowledge base for initial business routing and intent, then verify current implementation facts with CodeGraph, repository scans, and tests; if code and knowledge conflict, current code facts drive the task and the agent should recommend marking the affected knowledge document stale or needs-review. Code evidence and generated indexes can be refreshed automatically, but changes to business boundaries, terminology, design intent, context split/merge decisions, or confidence frontmatter require human confirmation or an explicit apply command. Design intent is recorded in capability documents by default, and only hard-to-reverse, non-obvious trade-offs are promoted to ADRs. CodeGraph and repository scans provide implementation evidence for each context, while design intent, boundaries, and architectural decisions are documented separately so the knowledge base explains why the business works this way, not just where code lives.
